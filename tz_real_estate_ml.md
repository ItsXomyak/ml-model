# Техническое задание

## Проект: ML-система оценки стоимости и определения класса квартир

## 1. Общие сведения

### 1.1. Назначение работы
Разработать ML-систему, решающую две связанные задачи на рынке недвижимости г. Астаны:
1. **Регрессия** — предсказание стоимости квартиры на основе её характеристик.
2. **Кластеризация** — автоматическое определение класса квартиры (эконом / комфорт / бизнес / премиум) без ручной разметки, методом k-means.

Система упаковывается в Docker-контейнер согласно требованиям MLOps-блока задания.

### 1.2. Цели проекта
- Закрыть все 8 заданий рубежного контроля (Блоки 1–4).
- Получить работающую модель, которую можно воспроизвести одной командой `docker run`.
- Продемонстрировать корректную работу с переобучением через регуляризацию.

### 1.3. Входные данные
Синтетический датасет квартир г. Астаны (~3000 записей), генерируемый скриптом `generate_data.py`. Реальный датасет не требуется — синтетика позволяет контролировать распределения и закладывать понятные зависимости для проверки модели.

### 1.4. Выходные результаты
- Обученная модель регрессии (сохраняется в `models/price_model.pkl`).
- Обученная модель кластеризации (сохраняется в `models/cluster_model.pkl`).
- Файл с метриками `metrics.json` (MSE, R², оптимальное K).
- Логи в stdout при запуске контейнера.

## 2. Структура датасета

Датасет содержит признаки одной квартиры в одной строке.

| Признак          | Тип       | Описание                                      | Диапазон / значения                                                         |
|------------------|-----------|-----------------------------------------------|------------------------------------------------------------------------------|
| `area`           | float     | Общая площадь, м²                             | 25 – 200                                                                     |
| `rooms`          | int       | Количество комнат                             | 1 – 5                                                                        |
| `floor`          | int       | Этаж квартиры                                 | 1 – 25                                                                       |
| `total_floors`   | int       | Этажей в доме                                 | 5 – 25                                                                       |
| `year_built`     | int       | Год постройки                                 | 1960 – 2025                                                                  |
| `district`       | category  | Район Астаны                                  | Есиль, Алматы, Сарыарка, Байконыр, Нура                                      |
| `material`       | category  | Материал стен                                 | панель, кирпич, монолит                                                      |
| `renovation`     | category  | Тип ремонта                                   | без ремонта, косметический, евро, дизайнерский                               |
| `dist_to_center` | float     | Расстояние до центра, км                      | 0.5 – 25                                                                     |
| `price`          | float     | **Целевая переменная** — цена в млн ₸         | 15 – 250                                                                     |

Цена в синтетике формируется как функция от признаков с добавлением шума для реалистичности.

## 3. Функциональные требования

### 3.1. Модуль генерации данных (`generate_data.py`)
- На входе: ничего (или параметр `--n` для размера выборки, по умолчанию 3000).
- На выходе: файл `data/apartments_astana.csv`.
- Должен использовать `numpy.random.seed(42)` для воспроизводимости.
- Цена должна реалистично зависеть от площади, района, ремонта и года постройки.

### 3.2. Главный скрипт (`train.py`)
Скрипт обязан выполнять следующие шаги последовательно:

**Шаг 1. Загрузка данных**
Чтение `data/apartments_astana.csv` через pandas.

**Шаг 2. Препроцессинг**
- Кодирование категориальных признаков через `OneHotEncoder` или `pd.get_dummies`.
- Масштабирование числовых признаков через `StandardScaler` (для Lasso/Ridge обязательно).
- Объединение в `ColumnTransformer` или `Pipeline`.

**Шаг 3. Разделение данных (Задание 2)**
- Пропорция: 60% train / 20% val / 20% test.
- `random_state=42` для воспроизводимости.
- Двухступенчатый split через `train_test_split`.
- Для регрессии стратификация по биннингу цены (опционально), для классификации — `stratify=y`.

**Шаг 4. Базовая модель регрессии (Задание 3)**
- Алгоритм: `LinearRegression` или `RandomForestRegressor`.
- Расчёт метрик на val: **MSE и R²**.
- Печать метрик в консоль.

**Шаг 5. Регуляризация (Задание 5)**
- Обучение `Lasso` (L1) и `Ridge` (L2) поверх того же препроцессинга.
- Подбор `alpha` через перебор по валидационной выборке.
- Сравнение метрик с базовой моделью.

**Шаг 6. Кластеризация (Задание 4)**
- Подготовка фичей для k-means: цена за м², площадь, расстояние до центра, район (закодированный).
- **Elbow method**: перебор K от 1 до 10, расчёт inertia, поиск точки локтя.
- Обучение финального k-means с оптимальным K.
- Маппинг номеров кластеров на названия классов (эконом / комфорт / бизнес / премиум) по средней цене за м² внутри кластера.

**Шаг 7. Финальная оценка**
- Прогноз на тестовой выборке.
- Сохранение моделей через `joblib.dump`.
- Запись метрик в `metrics.json`.

### 3.3. Файл `theory.md`
Содержит письменные ответы на теоретические задания (на русском):
- **Задание 1** — разница между val и test, почему нельзя тюнить гиперпараметры по test.
- **Задание 5 (теория)** — сравнение L1 и L2, когда использовать Lasso.
- **Задание 6** — три конкретных шага при overfit (98% train / 72% val).

### 3.4. Dockerfile (Задание 7)

```dockerfile
WORKDIR /app

```

### 3.5. README.md (Задание 8)

```bash

```

## 4. Нефункциональные требования

### 4.1. Воспроизводимость
- `random_state=42` везде, где есть случайность (split, RandomForest, KMeans, генерация данных).
- Запуск `python train.py` дважды должен давать идентичные метрики с точностью до 4 знаков.

### 4.2. Качество кода
- Python 3.9+.
- PEP8 (строки до 100 символов).
- Все блоки логики обёрнуты в функции (`load_data`, `split_data`, `train_regression`, `train_clustering`, `evaluate`, `main`).
- В `train.py` есть `if __name__ == "__main__": main()`.
- Комментарии на русском языке к ключевым шагам.

### 4.3. Зависимости (`requirements.txt`)
```
pandas>=2.0
numpy>=1.24
scikit-learn>=1.3
joblib>=1.3
matplotlib>=3.7
```
matplotlib нужен только если будем сохранять график elbow в файл. Если не нужен — убрать.

### 4.4. Производительность
- Полный прогон `train.py` на синтетике из 3000 строк — не более 60 секунд на стандартном ноутбуке.
- Размер Docker-образа — не более 500 МБ.

### 4.5. Метрики качества модели (минимально приемлемые)
- **R² на test ≥ 0.85** для регрессии (т.к. синтетика управляемая, должно достигаться легко).
- **MSE на test** должен быть в разумных пределах (порядок единиц при цене в млн ₸).
- Разрыв между train и val метриками не должен превышать 10% (иначе triggers Задание 6).

## 9. Риски и ограничения

| Риск | Митигация |
|------|-----------|
| Синтетика слишком "идеальная" → R² близко к 1 | Добавить шум при генерации цены (gaussian noise ~10%). |
| Overfit на малой выборке | Использовать минимум 3000 строк, RandomForest с `max_depth`. |
| Docker-образ слишком большой | Использовать `python:3.9-slim`, флаг `--no-cache-dir`. |
| Elbow не показывает явный излом | Дополнить silhouette score как вторую метрику. |
